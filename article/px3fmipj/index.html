Bean的生命周期

约 1983 字大约 7 分钟

2025-03-12

"Bean的生命周期"指的是从创建到销毁的过程中，一个Bean所经历的所有阶段。

  1. 实例化： 首先，当IoC容器启动时，它会根据配置（如XML配置或注解）来确定需要创建哪些Bean实例。在这个阶段，仅仅是对象被实例化，并没有进行任何属性的设置。 构造器注入：在这种情况下，你定义一个包含所需依赖作为参数的构造函数。当Spring需要创建Bean的实例时，它会根据配置信息解析出合适的依赖，并将它们作为参数传递给构造函数。这样，在对象实例化的同时，它的依赖也被注入了。 因此，如果你完全依赖于构造器注入，那么在对象实例化完成后，其依赖关系就已经被正确设置了，这就不算是“空壳”。
  2. 属性填充（Population） 一旦Bean被实例化，Spring IoC容器会将配置文件中定义的属性值或者依赖注入到这个Bean中。这一步骤是通过setter方法或构造函数注入完成的。
  3. 初始化前： 在Bean的属性设置完成后，但在初始化之前，Spring允许我们使用BeanNameAware, BeanFactoryAware, 和 ApplicationContextAware接口提供的回调方法，让Bean能够知道自己的名字、所属的BeanFactory以及整个ApplicationContext。
  4. 初始化：
    • 如果实现了InitializingBean接口，那么其afterPropertiesSet()方法会被调用。
    • 另外，如果在配置文件中指定了init-method，那么指定的方法也会在此时被调用。
  5. 后初始化： 此时Bean已经完全准备好并可以使用了。但是，如果Bean实现了DisposableBean接口或配置文件中定义了destroy-method，那么在容器关闭时，这些方法将会被调用来清理资源。
  6. 销毁： 当应用上下文关闭时，Spring容器会调用那些实现了DisposableBean接口的Bean的destroy()方法，或者执行用户自定义的销毁方法（通过配置文件中的destroy-method属性指定），以便于释放占用的资源。

Spring还提供了灵活的方式来管理Bean的作用域（singleton, prototype等），这也会影响Bean的生命周期行为。 例如，singleton作用域下的Bean在整个应用程序的生命周期内都是一样的实例，而prototype作用域下的每个获取都会返回一个新的实例。

初始化：

  1. Aware接口回调：如果Bean实现了某些*.Aware接口（如BeanNameAware, BeanClassLoaderAware, BeanFactoryAware等），Spring容器会在适当的时候调用这些接口提供的相应方法。这使得Bean能够获取到一些与自身或其所在的环境相关的信息或对象。例如： 如果实现了BeanNameAware，则调用setBeanName(String name)方法，传入Bean的名字。 如果实现了BeanClassLoaderAware，则调用setBeanClassLoader(ClassLoader classLoader)方法，传入ClassLoader对象的实例。 如果实现了BeanFactoryAware，则调用setBeanFactory(BeanFactory beanFactory)方法，传入BeanFactory对象的实例。
  2. BeanPostProcessor前置处理：如果有任何实现了BeanPostProcessor接口的Bean存在，它们的postProcessBeforeInitialization()方法会被调用。这是给Bean提供额外修改的机会，比如可以在这里改变Bean的属性值或其他设置。
  3. 初始化回调：接下来，对于那些需要执行初始化逻辑的Bean，Spring提供了两种方式： 如果Bean实现了InitializingBean接口，则调用其afterPropertiesSet()方法。这是一个标准化的方式来指定初始化逻辑。 另外，你可以在Bean的配置中指定一个init-method，当Bean被初始化时，Spring会自动调用该方法。
  4. BeanPostProcessor后置处理：在所有上述初始化步骤完成后，如果存在实现了BeanPostProcessor接口的Bean，它们的postProcessAfterInitialization()方法将被调用。这一步通常用于在Bean完全初始化之后进行最后的调整或包装，例如AOP代理的创建。

BeanNameAware： Bean 在初始化过程中获取到自己在 Spring 容器中的名称

import org.springframework.beans.factory.BeanNameAware;
import org.springframework.stereotype.Component;

@Component("myBean")
public class MyBean implements BeanNameAware {

    private String beanName;

    @Override
    public void setBeanName(String name) {
        // 保存bean的名字
        this.beanName = name;
        System.out.println("当前Bean的名称是: " + beanName);
    }

    public void displayInfo() {
        System.out.println("展示信息，Bean名称为: " + beanName);
    }
}

BeanClassLoaderAware： Bean 在初始化过程中获取到加载它的 ClassLoader 实例

import org.springframework.beans.factory.BeanClassLoaderAware;
import org.springframework.stereotype.Component;

@Component
public class MyBeanClassLoaderAware implements BeanClassLoaderAware {

    private ClassLoader classLoader;

    @Override
    public void setBeanClassLoader(ClassLoader classLoader) {
        // 保存ClassLoader实例
        this.classLoader = classLoader;
        System.out.println("当前Bean的ClassLoader是: " + classLoader);
    }

    public void displayClassLoaderInfo() {
        if (classLoader != null) {
            System.out.println("展示信息，ClassLoader为: " + classLoader);
        } else {
            System.out.println("ClassLoader尚未设置");
        }
    }
}

当你需要动态加载类或资源文件时，可以使用获得的 ClassLoader 来加载这些类或资源。 在某些高级场景下，如自定义类加载策略、处理不同类加载器之间的隔离等情况下可能会需要直接操作 ClassLoader。

BeanFactoryAware： 允许 Bean 在初始化过程中获取到加载它的 BeanFactory 实例

import org.springframework.beans.BeansException;
import org.springframework.beans.factory.BeanFactory;
import org.springframework.beans.factory.BeanFactoryAware;
import org.springframework.stereotype.Component;

@Component
public class MyBeanFactoryAware implements BeanFactoryAware {

    private BeanFactory beanFactory;

    @Override
    public void setBeanFactory(BeanFactory beanFactory) throws BeansException {
        // 保存BeanFactory实例
        this.beanFactory = beanFactory;
        System.out.println("当前Bean的BeanFactory是: " + beanFactory);
    }

    public void displayBeanFactoryInfo() {
        if (beanFactory != null) {
            System.out.println("展示信息，BeanFactory为: " + beanFactory);
            // 可以使用beanFactory来获取其他Bean等操作
        } else {
            System.out.println("BeanFactory尚未设置");
        }
    }
}

BeanPostProcessor：

import org.springframework.beans.factory.config.BeanPostProcessor;
import org.springframework.stereotype.Component;

@Component
public class MyBeanPostProcessor implements BeanPostProcessor {

    @Override
    public Object postProcessBeforeInitialization(Object bean, String beanName) {
        System.out.println("Before initialization: " + beanName);
        // 可以在这里修改bean实例，但通常返回原始bean实例
        return bean;
    }

    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName) {
        System.out.println("After initialization: " + beanName);
        // 同样可以在这里修改bean实例，但通常返回原始bean实例
        return bean;
    }
}

InitializingBean：

  1. 实现接口
import org.springframework.beans.factory.InitializingBean;
import org.springframework.stereotype.Component;

@Component
public class MyInitializingBean implements InitializingBean {

    @Override
    public void afterPropertiesSet() throws Exception {
        // 在这里添加初始化逻辑
        System.out.println("MyInitializingBean 已经完成属性设置，现在可以进行初始化工作了。");
    }

    // 你可以在这里定义其他方法或属性
}
  1. PostConstruct 注解
import javax.annotation.PostConstruct;

public class MyBean {

    @PostConstruct
    public void init() {
        // 初始化逻辑
    }
}
  1. 在 XML 中指定 init-method：
<bean id="myBean" class="com.example.MyBean" init-method="init"/>

销毁Bean的过程：

  1. 实现DisposableBean
import org.springframework.beans.factory.DisposableBean;

public class MyBean implements DisposableBean {
   @Override
   public void destroy() throws Exception {
    // 清理资源，如关闭数据库连接等
   }
}
  1. @PreDestroy 注解
import javax.annotation.PreDestroy;

public class MyBean {
    @PreDestroy
    public void cleanup() {
       // 清理资源，如关闭数据库连接等
    }
}
  1. 在XML配置中指定destroy-method属性
<bean id="myBean" class="com.example.MyBean" destroy-method="cleanup"/>

Bean进入销毁流程的方式：

  1. Bean进入销毁流程的方式 对于单例作用域的Bean，它们会在Spring应用上下文关闭时自动被销毁。这意味着你不需要手动去销毁这些Bean，Spring会负责调用它们的销毁方法（如果已经定义）。 关闭Spring应用上下文：你可以通过关闭包含该Bean的应用上下文来触发Bean的销毁。例如，在一个Web应用程序中，当Servlet容器停止时，Spring的ContextLoaderListener或DispatcherServlet会自动关闭相应的应用上下文。
ApplicationContext context = ... // 获取你的应用上下文
context.close(); // 这将触发所有单例Bean的销毁方法
  1. 非单例(Prototype)作用域的Bean

对于非单例作用域的Bean，Spring不会自动管理它们的生命周期。这意味着你需要手动管理这些Bean的创建和销毁。 如果你为prototype Bean指定了destroy-method，你需要自己确保在适当的时候调用这个方法。

MyPrototypeBean myPrototypeBean = context.getBean(MyPrototypeBean.class);
// 使用完之后
myPrototypeBean.cleanup(); // 假设cleanup是你定义的销毁方法
  1. 自定义作用域的Bean 对于自定义作用域的Bean（比如request、session等），它们的销毁时机由其作用域决定。例如，对于绑定到HTTP请求的Bean，当请求结束时，Spring会自动销毁该Bean。
